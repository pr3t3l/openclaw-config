05The 3-Layer Memory System

AI agents forget everything between conversations. The memory system gives Robotin continuity — it remembers what happened yesterday, what decisions were made last month, and can search across its entire history.

Layer 1: Daily Notes
Files named by date (memory/YYYY-MM-DD.md) — raw logs of what happened each day. Today's and yesterday's notes are loaded automatically at the start of every session. Written by the agent during conversations.
Layer 2: Long-Term Memory
A curated file (MEMORY.md) containing decisions, preferences, key facts, and system state. Loaded only in main sessions — not in sub-agent or heartbeat sessions. Manually maintained.
Layer 3: Semantic Search (QMD)
QMD 2.0.1 + Bun runtime — semantic search across all memory files. Updates every 5 minutes, returns up to 6 results within a 4-second timeout. Only available in direct chats, not sub-agent contexts.

When the agent's context window gets full, a compaction process kicks in. But before any context is pruned, a memoryFlush at 6,000 tokens saves the current conversation state to daily notes. This prevents the agent from "forgetting" what it was doing mid-task.

06Model Routing: Two Permission Layers

Adding a new AI model to the system requires updating two places in the configuration — and forgetting either one means the agent can't use it. The first layer (models.providers.litellm.models) declares which models exist and what capabilities they have. The second layer (agents.defaults.models) authorizes specific models for agent use.

This two-layer design is intentional — it prevents agents from accidentally using expensive or inappropriate models. But it's also a common source of confusion: if you add a model to LiteLLM but forget to authorize it in openclaw.json, the agent will fail silently.

02The Zero-Templates Revolution

Before V9, the pipeline used Handlebars templates — pre-built HTML skeletons where content was inserted into fixed slots. This worked for simple documents but broke down fast. A police interrogation transcript needs a completely different layout than a social media screenshot or a newspaper front page. Templates were too rigid. When the AI wanted to create a floor plan diagram or a surveillance camera timeline, the templates couldn't handle it.

V9 eliminated templates entirely. Now Claude receives the content (a _content.md file with text and [IMAGE:] placement tags) plus a design brief, and generates complete standalone HTML from scratch — CSS, layout, typography, aging effects, stamps, everything. Then headless Chromium converts it to PDF. Every document is unique. A forensic toxicology report looks different from an Instagram DM screenshot, which looks different from a handwritten note.

"Quality is designed in the clue catalog, not QA'd afterwards."

The pipeline uses a two-tier prompt system for rendering: text-heavy documents (transcripts, court orders, witness statements) get a minimal styling prompt, while visual documents (social media mockups, floor plans, POI sheets) get a full design system prompt with detailed visual instructions. One strict rule applies to all documents: backgrounds must always be white or light — these are meant to look like real paper documents, not dark-themed web pages.

03The Spawn Problem (and Its Solution)

OpenClaw's built-in sub-agent spawner (sessions_spawn) is broken for file-writing tasks. Sub-agents complete with "(no output)" and zero tokens — the work vanishes. This was confirmed broken 6+ times across multiple cases and phases. It's an upstream platform limitation that can't be fixed.

The replacement: three spawn scripts that bypass OpenClaw's spawner entirely. Each script calls the AI API directly via streaming curl (Python's requests library fails in WSL for long API calls — a hard-learned lesson), and the script writes the output files, not the sub-agent. This means cost tracking works, file writing is reliable, and timeouts are controllable.

spawn_agent.py
Generic content generation. Used for Art Director, Experience Designer, Production Engine, and QA phases. Calls the API, receives the response, writes files to disk.
spawn_narrative.py
Specialized for the Narrative Architect. Supports --phase plan (case plan) and --phase catalog --envelope X (per-envelope clue catalog generation).
spawn_images.py

02La Revolución Zero-Templates

Antes de V9, el pipeline usaba plantillas Handlebars — esqueletos HTML pre-construidos donde el contenido se insertaba en slots fijos. Esto funcionaba para documentos simples pero se rompía rápido. Una transcripción de interrogatorio policial necesita un layout completamente diferente a una captura de redes sociales o una portada de periódico. Las plantillas eran demasiado rígidas.

V9 eliminó las plantillas por completo. Ahora Claude recibe el contenido (un archivo _content.md con texto y tags [IMAGE:]) más un brief de diseño, y genera HTML standalone completo desde cero — CSS, layout, tipografía, efectos de envejecimiento, sellos, todo. Luego Chromium headless lo convierte a PDF. Cada documento es único.

"La calidad se diseña en el catálogo de pistas, no se QA-ea después."

El pipeline usa un sistema de prompts de dos niveles: documentos de texto pesado (transcripciones, órdenes judiciales) obtienen un prompt de estilo minimal, mientras que documentos visuales (mockups de redes sociales, planos, fichas POI) obtienen un prompt completo del sistema de diseño. Una regla estricta aplica a todos: los fondos deben ser siempre blancos o claros — están diseñados para parecer documentos de papel reales.

02Three Phases: Clarify → Design → Build Plan

Every idea goes through three phases. Each produces validated JSON artifacts, and each ends with a human gate where you review the output and decide whether to proceed, adjust, or stop. The system never auto-approves — early versions did, and quality issues went unnoticed until they compounded.

The Gap Finder adversarially identifies what's missing and assigns a readiness score (0-100) — a score of 38 means "this idea has 12 major gaps." The Scope Framer proposes three tiers: MVP, Standard, and Advanced. The system supports up to 5 rounds of clarifying questions before forcing the idea to "ready" status.

03The 3-Round Multi-Model Debate

Not every idea needs a debate. Simple ideas get a single model producing one proposal. Complex ideas (multi-component systems) get two models debating. Critical ideas (infrastructure, high-cost, novel architectures) get three models plus a Red Team attack and a judge. The principle: debate is conditional, not ritual — simple ideas don't need three models arguing about them.

Round 1: Propose
Each model produces an independent architecture proposal. No cross-contamination — each thinks from its own perspective and training biases.
Round 2: Cross-Critique
Each model sees the others' proposals, critiques them specifically ("your caching strategy ignores X"), and revises its own. Best ideas propagate across models.
Round 3: Consolidate
Synthesize the best elements into a unified proposal. For critical ideas: Red Team then attacks it looking for 10+ flaws. A judge model evaluates the final result.

04GPT's 7 Principles

These emerged from GPT-5.4's review of Claude's first design and became the Planner's operating philosophy. They're not just design rules — they're lessons from what went wrong in the 9 versions of the Pipeline.

P-01: Gap Finder first. Understand what's missing before designing anything. The Pipeline's V6 failure (58 schema errors) happened because gaps weren't identified upfront.

P-02: Data flow before architecture. Define what data moves before deciding who moves it.

P-03: Contracts before code. Agree on JSON schemas before writing implementation. Components that disagree on data formats break at integration time.
P-04: Debate is conditional, not ritual. Simple ideas don't need 3-model debate — it's a waste of money and time.

P-05: Cost estimator is a script. Math is deterministic. Don't waste tokens on arithmetic.

P-06: Human gates persist decisions. No auto-approvals. Decisions are recorded and immutable.

P-07: Model routing in config, not code. One file to update when models change.

A product-agnostic AI engine that generates, verifies, and optimizes marketing content for any product. Give it a product brief — what you sell, who buys it, what's true about it — and it produces 30 verified content pieces per week, learns from performance, and improves every cycle. 01 How It Works: Three Layers

02 The Product Brief: How Any Product Plugs In

Every product that uses the Marketing System starts with a product brief — a structured JSON file (product_brief.json) that tells the system what the product is, what can be said about it, and what must never be claimed. This file is the single source of truth that all content generators read from, and the Claim Linter verifies against.

A product brief contains three sections: verified facts (claims the generators are allowed to make), allowed claims (softer claims that are permissible with exact wording), and forbidden claims (things that must be blocked if the AI generates them). When you add a new product, you write this brief, run the Strategy Layer, and the system adapts everything — personas, keywords, content, and quality checks — to that product.

EXAMPLE — Current product: Declassified Cases
Verified Facts — the ONLY claims generators can use
✓"Instant digital download — play tonight" — true and verifiable on the store
✓"Starts at $19.99 per case" — real price listed on declassified.shop
✓"15-25 documents per case: police files, forensic reports, witness interviews" — actual product contents
✓"Designed for groups over video calls" — real feature of the product
Forbidden Claims — hard-blocked if detected
✗Fake testimonials — "María says it changed her weekends" (person doesn't exist, no real customers yet)
✗Invented statistics — "95% of players recommend it" (no survey was ever conducted)
✗Fake awards or press — "Award-winning experience" (no awards exist)
✗Competitor disparagement — "Better than [competitor]" (unverifiable comparative claim)
✗Urgency fabrication — "Only 5 left!" (digital product = unlimited supply)
Allowed Claims — permitted but must use exact wording
~"New case every week" — true at current cadence, subject to change
~"Perfect for date night" — subjective but matches a real use case
~"Growing catalog" — true as long as production continues

For a different product — say, artisan candles — the brief would contain entirely different facts ("hand-poured soy wax", "$24.99", "40-hour burn time"), different forbidden claims ("all-natural" if some fragrance oils are synthetic), and different personas. The system doesn't care what you sell. It cares that every claim is provably true.

03 The Strategy Layer: What It Produces

The Strategy Layer generates five foundational artifacts that define everything downstream: who the audience is, how to position against competitors, what keywords to target, what creative angles to use, and which channels to prioritize for each persona. These five files live in a versioned directory per product, and the Weekly Layer reads from them automatically.

Buyer Personas
5-10 segments defined by use case, motivation, preferred channel, and emotional trigger. Each persona gets tailored messaging in the Weekly Layer.
SEO Keywords
60-100 keywords distributed across pillar pages. Each maps to a persona and a content angle for organic search targeting.

25 angles approaching the product from different emotional triggers — curiosity, urgency, social proof, exclusivity, nostalgia. Prevents content repetition across weeks.

04 The Creative Engine: 6 Lanes

This prevents the #1 enemy of paid social content: creative fatigue, where audiences stop responding because they've seen the same angle too many times. Research from 30+ DTC marketing sources showed that 15-30 ad variants per week are needed to maintain engagement — the 6-lane system produces exactly that level of structured variation.

The 12 video scripts (2 per lane) are short-form scripts for Reels and TikTok, designed to hook within the first 3 seconds. The 12 ad copies target the persona groups defined in the strategy — each with messaging that speaks to their specific motivation. The 6 emails use three behavioral sequences: curiosity (create intrigue), urgency (time-limited hooks), and social proof (show others engaging with the product).

05 Zero Fabrications: The Quality Gate

The first time the system generated content for a real product, the quality checks caught 24 critical problems. The AI was confidently inventing things: fake customer testimonials, made-up statistics, nonexistent awards. These aren't minor copy edits — a fake testimonial published publicly is a legal liability and a trust destroyer for any brand.

"If it can't be proven from the product brief, it doesn't exist."

The Claim Linter is a Python script that uses 16 pattern-matching rules and 7 cross-reference checks to scan every piece of generated content against the product brief. It looks for things like:

What the linter catches
✗Quoted testimonials — any "Customer X says..." when no real customers exist yet
✗Percentage statistics — "95% agree" when no survey or study has been conducted
✗Superlative claims — "the best", "the only", "#1" without verifiable source
✗Award or press references — "featured in...", "award-winning" that don't exist
✗Artificial scarcity — "limited stock" on a digital/unlimited product
✗Claims not in verified_facts — any product feature or benefit not explicitly listed in the brief

If any critical issue is found, the pipeline stops completely — the AI quality reviewer doesn't even run. This is fail-closed design: when in doubt, block everything. The system published zero fabricated claims after this was implemented.

The Quality Reviewer is an AI-powered second pass that evaluates strategic alignment: is the content targeting the right persona? Does the emotional tone match the creative lane? Is the call-to-action clear? This runs on Claude Sonnet because GPT timed out when processing the full context (30 content pieces plus strategy documents). Critical findings block the pipeline; warnings are logged but don't stop publication.

El Sistema de Marketing

MOTOR DE CONTENIDO AUTOMATIZADO EN 3 CAPAS · ESTRATEGIA + SEMANAL + CRECIMIENTO
Un motor de IA agnóstico al producto que genera, verifica y optimiza contenido de marketing para cualquier producto. Dale un product brief — qué vendes, quién lo compra, qué es verdad sobre el producto — y produce 30 piezas de contenido verificadas por semana, aprende del rendimiento, y mejora cada ciclo. Actualmente impulsa Declassified Cases. Mañana podría impulsar velas, carros, SaaS, o lo que sea.
01Cómo Funciona: Tres Capas

El sistema tiene tres capas que trabajan en ciclo. La Capa de Estrategia se ejecuta una vez por producto y define a quién hablarle y qué decir. La Capa Semanal genera el contenido real cada semana. La Capa de Crecimiento mide qué funcionó, diagnostica por qué, y alimenta optimización de vuelta a la siguiente semana. Un cuarto componente — el Runtime — coordina todo y aplica una regla que no puede anularse: marketing nunca se ejecuta sin una estrategia válida y aprobada.

Todo está organizado alrededor de un product_id. Para mercadear un producto nuevo, creas un product_brief.json (describiendo el producto, sus hechos verificados y sus restricciones) y ejecutas la capa de estrategia. Las capas semanal y de crecimiento se adaptan automáticamente — el mismo motor, diferente producto.

02El Product Brief: Cómo se Conecta Cualquier Producto

Cada producto que usa el Marketing System comienza con un product brief — un archivo JSON estructurado (product_brief.json) que le dice al sistema qué es el producto, qué se puede decir sobre él, y qué nunca debe afirmarse. Este archivo es la única fuente de verdad que todos los generadores de contenido leen, y contra la cual el Claim Linter verifica.

Un product brief contiene tres secciones: hechos verificados (claims que los generadores pueden hacer), claims permitidos (claims más suaves permisibles con redacción exacta), y claims prohibidos (cosas que deben bloquearse si la IA las genera). Cuando agregas un producto nuevo, escribes este brief, ejecutas la Capa de Estrategia, y el sistema adapta todo.

EJEMPLO — Producto actual: Declassified Cases
Hechos Verificados — los ÚNICOS claims que los generadores pueden usar
✓"Descarga digital instantánea — juega esta noche" — verdadero y verificable en la tienda
✓"Desde $19.99 por caso" — precio real en declassified.shop
✓"15-25 documentos por caso: archivos policiales, reportes forenses, entrevistas" — contenido real
✓"Diseñado para grupos en videollamada" — característica real
Claims Prohibidos — bloqueados si se detectan
✗Testimonios falsos — "María dice que cambió sus fines de semana" (persona no existe)
✗Estadísticas inventadas — "95% de jugadores lo recomiendan" (ninguna encuesta se hizo)
✗Premios o prensa falsos — "Experiencia premiada" (no existen premios)
✗Menosprecio de competidores — "Mejor que [competidor]" (claim comparativo no verificable)
✗Urgencia fabricada — "¡Solo quedan 5!" (producto digital = stock ilimitado)
Claims Permitidos — con redacción exacta
~"Nuevo caso cada semana" — verdad a la cadencia actual
~"Perfecto para noche de cita" — subjetivo pero coincide con caso de uso
~"Catálogo creciente" — verdad mientras la producción continúe

Para un producto diferente — digamos, velas artesanales — el brief contendría hechos totalmente diferentes ("cera de soja vertida a mano", "$24.99", "40 horas de quemado"), claims prohibidos diferentes ("100% natural" si algunas fragancias son sintéticas), y personas diferentes. Al sistema no le importa qué vendes. Le importa que cada claim sea comprobablemente verdadero.

04El Motor Creativo: 6 Carriles

El contenido no se genera al azar. Cada semana sigue 6 carriles creativos — cada carril apunta a un trigger emocional específico. Esto previene el enemigo #1 del contenido social pagado: fatiga creativa, donde las audiencias dejan de responder porque han visto el mismo ángulo demasiadas veces. La investigación de 30+ fuentes de marketing DTC mostró que se necesitan 15-30 variantes de anuncios por semana.

Los 12 scripts de video (2 por carril) son scripts cortos para Reels y TikTok, diseñados para enganchar en los primeros 3 segundos. Los 12 copys de anuncios apuntan a los grupos de personas definidos en la estrategia. Los 6 emails usan tres secuencias conductuales: curiosidad (crear intriga), urgencia (ganchos de tiempo limitado), y prueba social (mostrar otros usando el producto).

05Cero Fabricaciones: La Puerta de Calidad

La primera vez que el sistema generó contenido para un producto real, las verificaciones de calidad capturaron 24 problemas críticos. La IA estaba inventando con confianza: testimonios falsos de clientes, estadísticas inventadas, premios inexistentes. Estos no son ediciones menores de copy — un testimonio falso publicado públicamente es una responsabilidad legal.

"Si no se puede probar desde el product brief, no existe."

El Claim Linter es un script Python que usa 16 reglas de coincidencia de patrones y 7 verificaciones cruzadas para escanear cada pieza de contenido generado contra el product brief. Busca cosas como:

Qué captura el linter
✗Testimonios citados — cualquier "El cliente X dice..." cuando no existen clientes reales
✗Estadísticas porcentuales — "95% está de acuerdo" sin encuesta ni estudio
✗Claims superlativos — "el mejor", "el único", "#1" sin fuente verificable
✗Referencias a premios o prensa — "destacado en...", "premiado" que no existen
✗Escasez artificial — "stock limitado" en un producto digital/ilimitado
✗Claims no en verified_facts — cualquier característica o beneficio no listado en el brief

Si se encuentra cualquier problema crítico, el pipeline se detiene completamente. Esto es diseño fail-closed: ante la duda, bloquear todo. El sistema publicó cero claims fabricados después de implementar esto.

El Quality Reviewer es una segunda pasada potenciada por IA que evalúa alineación estratégica: ¿el contenido apunta a la persona correcta? ¿El tono emocional coincide con el carril creativo? ¿El call-to-action es claro? Esto corre en Claude Sonnet porque GPT agotó tiempo al procesar el contexto completo (30 piezas de contenido más documentos de estrategia).

06Growth Intelligence: El Loop de Retroalimentación
"Mide antes de opinar. Diagnostica antes de cambiar."

Growth Intelligence se separa en dos niveles. Optimización táctica (ajustes semana-a-semana en copy, timing y targeting) sucede cada semana. Optimización estratégica (posicionamiento de mercado, pricing, cambios de audiencia) sucede trimestralmente, o se dispara automáticamente cuando umbrales duros se cruzan.

El loop produce tres salidas. Acciones de optimización se inyectan directamente en los prompts de generación de contenido de la siguiente semana. Patrones de knowledge base se acumulan con el tiempo — patrones confirmados se usan automáticamente, los tentativos se surfean como sugerencias. Alertas de estrategia pueden disparar invalidación de la estrategia actual.

CASE FILE — FINANCE TRACKER

Robotin Finance

PERSONAL EXPENSE TRACKING · AIRBNB TAX DEDUCTIONS · TELEGRAM
A personal finance tracker that lives inside Telegram. Send a photo of a receipt or type "$15.50 Uber" — AI parses it, categorizes it, logs it to Google Sheets, and monitors your budget. The differentiator: it automatically flags expenses that might be tax-deductible for rental property owners. Built as a skill inside the CEO agent (Robotin), with a commercialization path toward a standalone SaaS product.
01How It Works: From Receipt to Insight
📸 Send a receipt or text
Take a photo of a receipt or type "$15.50 Uber" in Telegram. The Rule Engine checks first — if it recognizes the merchant (Walmart, Uber, Netflix), it auto-categorizes at $0 cost without calling AI. If the merchant is unknown, the Expense Parser uses AI to extract merchant, amount, date, category, and individual items.
✅ Confirm and log
Robotin shows the parsed result: "Walmart $87.43 Groceries — ¿Confirmar?" You tap yes or correct it. The Transaction Logger writes the confirmed transaction to Google Sheets automatically.
📊 Daily cashflow
Every morning at 7:30 AM (weekdays), you get: "Safe to spend today: $42." The Daily Cashflow module calculates this as: remaining budget minus upcoming bills, divided by days until next payday.
⚠️ Budget alerts
When spending hits 80% of a category budget: "Shopping: $120 of $150 used (80%). 11 days left in month." Plus payment reminders 3 days before due dates for bills and loans.
COST: Rule Engine matches = $0 per parse. AI parsing (unknown merchants) = ~$0.001-$0.003 for text, ~$0.005-$0.01 for photos. Monthly AI analyst report = ~$0.01. Everything else is deterministic and free.
02The 8 Modules

The tracker is built as 8 modules, each with a clear responsibility. The Rule Engine runs first (cheapest path), then AI parsing if needed, then logging, then monitoring. Monthly analysis and bank reconciliation run on their own schedules.

01
Rule Engine
Auto-categorize known merchants via pattern matching. If "Walmart" → Groceries. If "Netflix" → Subscriptions_Other. No AI needed, $0 per parse. Rules stored in rules.json.
02
Expense Parser
AI-powered receipt parsing for unknown merchants. Handles both photos (vision model) and text input. Extracts merchant name, amount, date, category, and individual line items.
03
Transaction Logger
Writes confirmed transactions to Google Sheets via gspread API. Uses batch writes to avoid Google's 429 rate limit errors.
04
Budget Monitor
Checks spending against category budgets. Alerts via Telegram when spending approaches configurable thresholds (default: 80%).
05
Payment Reminder
Checks payment calendar for bills due within 3 days. Sends Telegram alerts with amount, creditor, and account.
06
Daily Cashflow
Calculates "safe to spend today" = (remaining budget - upcoming bills) / days until payday. Runs weekday mornings at 7:30 AM.
07
Monthly Analyst
AI-generated month-end report: spending trends, budget vs actual, category breakdown, and recommendations. The only module besides the parser that uses AI.
08
Reconciliation (v2)
Matches logged receipts against bank CSV exports. Multi-month support, false positive elimination, signed amounts in a dedicated Cashflow_Ledger tab. Supports Wells Fargo checking CSV format.

A 9th module has been designed but not implemented: Batch Receipt Processor. It processes 17+ receipt links at once (Walmart, Target, etc.), replacing generic CSV-imported transactions with detailed line-item breakdowns. Includes deduplication and batch Airbnb confirmation for cleaning/supply items.

03Airbnb Tax Deduction Tracking

This is the feature that differentiates this tracker from every other expense app. For rental property owners, certain expenses — cleaning supplies, linens, repairs, a portion of utilities — are tax-deductible on Schedule E. Most expense trackers don't handle this. Robotin Finance does.

When the parser detects items that could be Airbnb-related (cleaning supplies at Home Depot, property insurance, utility bills), it asks: "Personal or Airbnb?" The answer is tagged in Google Sheets with two fields: tax_deductible (true/false/null) and tax_category (airbnb_supply, airbnb_repair, airbnb_utility, airbnb_insurance, or personal). For utility bills, it asks the split percentage — how much is for the rental vs personal use.

At tax time, the command "Tax summary" generates a total deductible amount by category. "Tax export" produces a CSV formatted for Schedule E that can be sent directly to an accountant.

STATUS: Designed and spec'd in the upgrade document. Not yet implemented. This is the core feature for the SaaS commercialization path.

Un rastreador de finanzas personales que vive dentro de Telegram. Envía una foto de un recibo o escribe "$15.50 Uber" — la IA lo parsea, categoriza, registra en Google Sheets y monitorea tu presupuesto. El diferenciador: automáticamente señala gastos que podrían ser deducibles de impuestos para propietarios de propiedades de alquiler. Construido como un skill dentro del agente CEO (Robotin), con un camino de comercialización hacia un producto SaaS independiente.
01Cómo Funciona: Del Recibo al Insight
📸 Envía un recibo o texto
Toma una foto de un recibo o escribe "$15.50 Uber" en Telegram. El Rule Engine verifica primero — si reconoce el comercio (Walmart, Uber, Netflix), auto-categoriza a $0 sin llamar IA. Si el comercio es desconocido, el Expense Parser usa IA para extraer comercio, monto, fecha, categoría e ítems individuales.
✅ Confirmar y registrar
Robotin muestra el resultado parseado: "Walmart $87.43 Groceries — ¿Confirmar?" Tocas sí o corriges. El Transaction Logger escribe la transacción confirmada en Google Sheets automáticamente.
📊 Cashflow diario
Cada mañana a las 7:30 AM (días laborales): "Seguro para gastar hoy: $42." El módulo Daily Cashflow calcula esto como: presupuesto restante menos facturas pendientes, dividido por días hasta el próximo pago.
⚠️ Alertas de presupuesto
Cuando el gasto alcanza 80% del presupuesto de una categoría: "Shopping: $120 de $150 usados (80%). 11 días restantes en el mes." Más recordatorios de pago 3 días antes de fechas de vencimiento.
COSTO: Coincidencias Rule Engine = $0 por parse. Parseo IA (comercios desconocidos) = ~$0.001-$0.003 texto, ~$0.005-$0.01 fotos. Reporte mensual IA = ~$0.01. Todo lo demás es determinista y gratis.
02Los 8 Módulos

El tracker está construido como 8 módulos, cada uno con una responsabilidad clara. El Rule Engine corre primero (camino más barato), luego parseo IA si es necesario, luego registro, luego monitoreo. El análisis mensual y la reconciliación bancaria corren en sus propios horarios.

01
Rule Engine
Auto-categorizar comercios conocidos vía coincidencia de patrones. Si "Walmart" → Groceries. Si "Netflix" → Subscriptions_Other. Sin IA, $0. Reglas almacenadas en rules.json.
02
Expense Parser
Parseo de recibos potenciado por IA para comercios desconocidos. Maneja fotos (modelo de visión) y texto. Extrae comercio, monto, fecha, categoría e ítems de línea.
03
Transaction Logger
Escribe transacciones confirmadas a Google Sheets vía API gspread. Usa escrituras batch para evitar errores de rate limit 429 de Google.
04
Budget Monitor
Verifica gasto contra presupuestos de categoría. Alerta vía Telegram cuando el gasto se acerca a umbrales configurables (default: 80%).
05
Payment Reminder
Verifica calendario de pagos para facturas que vencen en los próximos 3 días. Envía alertas por Telegram.
06
Daily Cashflow
Calcula "seguro para gastar hoy" = (presupuesto restante - facturas pendientes) / días hasta día de pago. Corre mañanas de lunes a viernes a las 7:30 AM.
07
Monthly Analyst
Reporte de fin de mes generado por IA: tendencias de gasto, presupuesto vs real, desglose por categoría y recomendaciones.
08
Reconciliation (v2)
Coteja recibos registrados contra exportaciones CSV bancarias. Soporte multi-mes, eliminación de falsos positivos, montos con signo en tab Cashflow_Ledger dedicado.

Un 9no módulo ha sido diseñado pero no implementado: Batch Receipt Processor. Procesa 17+ links de recibos a la vez, reemplazando transacciones genéricas importadas por CSV con desgloses detallados por ítem.

03Rastreo de Deducciones Fiscales Airbnb

Esta es la característica que diferencia este tracker de cualquier otra app de gastos. Para propietarios de propiedades de alquiler, ciertos gastos — suministros de limpieza, ropa de cama, reparaciones, una porción de servicios — son deducibles en el Schedule E. La mayoría de trackers de gastos no manejan esto. Robotin Finance sí.

Cuando el parser detecta ítems que podrían estar relacionados con Airbnb (suministros de limpieza en Home Depot, seguro de propiedad, facturas de servicios), pregunta: "¿Personal o Airbnb?" La respuesta se etiqueta en Google Sheets con dos campos: tax_deductible (true/false/null) y tax_category (airbnb_supply, airbnb_repair, airbnb_utility, airbnb_insurance, o personal). Para facturas de servicios, pregunta el porcentaje de división.

En época de impuestos, el comando "Tax summary" genera un total deducible por categoría. "Tax export" produce un CSV formateado para Schedule E que se puede enviar directamente al contador.

ESTADO: Diseñado y especificado en el documento de upgrade. Aún no implementado. Esta es la característica core para el camino de comercialización SaaS.
