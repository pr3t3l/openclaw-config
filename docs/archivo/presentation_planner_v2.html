Meta-Workflow Planner

THE WORKFLOW THAT BUILDS WORKFLOWS · V3.1
The Declassified Cases Pipeline took 9 versions to get right. The Planner exists so that never happens again. It takes a raw, unstructured idea — "I need a system that does X" — validates it through multi-model debate, identifies gaps before any code is written, and produces a buildable implementation plan. When the architecture docs are already solid, skip it and go straight to code. When the idea is fuzzy, this is where it gets sharp.
01Designed by 4 AI Models

The Planner was designed through the same cross-model collaboration it now facilitates. Claude Opus produced the first version: 10 agents, 4 phases, always-debate architecture. GPT-5.4 challenged it with a counter-proposal and 7 design principles — the most impactful being P-01: "Close gaps first, then design." Instead of jumping straight to architecture, find what's missing from the idea first.

Claude merged the best of both into v2. GPT reviewed v2 and added 4 more fixes (no symlinks, config-based routing, gate persistence, proper error handling). Then Gemini 3.1 Pro and MiniMax M2.7 contributed input on edge cases and failure modes. The final v3 — a 966-line master plan — was locked and built in 4 phases over a single day.

"First close gaps, then design." — GPT-5.4's key insight that became Principle P-01
02Three Phases: Clarify → Design → Build Plan

Every idea goes through three phases. Each produces validated JSON artifacts, and each ends with a human gate where you review the output and decide whether to proceed, adjust, or stop. The system never auto-approves — early versions did, and quality issues went unnoticed until they compounded.

Phase A — Clarify
The Intake Analyst extracts structure from the raw idea. The Gap Finder adversarially identifies what's missing and assigns a readiness score (0-100) — a score of 38 means "this idea has 12 major gaps." The Scope Framer proposes three tiers: MVP, Standard, and Advanced. The system supports up to 5 rounds of clarifying questions before forcing the idea to "ready" status.

Phase B — Design
The Data Flow Mapper defines all artifacts: what data moves, who produces it, who consumes it. The Contract Designer creates JSON schemas for every artifact — ensuring all components agree on data formats before code exists. The Architecture Planner triggers a multi-model debate (see next section) to design the system architecture. For critical ideas, a Red Team attacks the result looking for flaws.

Phase C — Buildability
The Implementation Planner creates a phased build plan with specific tests for each phase. The Cost Estimator is a pure Python script (not an AI call) that calculates expected costs deterministically — math doesn't need tokens. The Lessons Validator checks the plan against a library of past mistakes (things like "never use symlinks across workspaces" or "SKILL.md must include JSON skeletons").

03The 3-Round Multi-Model Debate

Not every idea needs a debate. Simple ideas get a single model producing one proposal. Complex ideas (multi-component systems) get two models debating. Critical ideas (infrastructure, high-cost, novel architectures) get three models plus a Red Team attack and a judge. The principle: debate is conditional, not ritual — simple ideas don't need three models arguing about them.

Round 1: Propose
Each model produces an independent architecture proposal. No cross-contamination — each thinks from its own perspective and training biases.
Round 2: Cross-Critique
Each model sees the others' proposals, critiques them specifically ("your caching strategy ignores X"), and revises its own. Best ideas propagate across models.
Round 3: Consolidate
Synthesize the best elements into a unified proposal. For critical ideas: Red Team then attacks it looking for 10+ flaws. A judge model evaluates the final result.

04GPT's 7 Principles

These emerged from GPT-5.4's review of Claude's first design and became the Planner's operating philosophy. They're not just design rules — they're lessons from what went wrong in the 9 versions of the Pipeline.

P-01: Gap Finder first. Understand what's missing before designing anything. The Pipeline's V6 failure (58 schema errors) happened because gaps weren't identified upfront.

P-02: Data flow before architecture. Define what data moves before deciding who moves it.

P-03: Contracts before code. Agree on JSON schemas before writing implementation. Components that disagree on data formats break at integration time.
P-04: Debate is conditional, not ritual. Simple ideas don't need 3-model debate — it's a waste of money and time.

P-05: Cost estimator is a script. Math is deterministic. Don't waste tokens on arithmetic.

P-06: Human gates persist decisions. No auto-approvals. Decisions are recorded and immutable.

P-07: Model routing in config, not code. One file to update when models change.

06How to Use It

The Planner has its own dedicated Telegram bot (@Super_Workflow_Creator_bot). Send it a message describing your idea — in natural language, no special format required — and it starts Phase A. The CEO bot (Robotin) can also route "Planifica:" prefixed messages to the Planner, though direct access is preferred because the CEO's fallback model sometimes ignores complex routing.

When to use it: Raw, unstructured ideas that need validation. "I want to build an automated invoicing system" — the Planner will find what's missing, design the architecture, and produce a buildable plan. When to skip it: When you already have detailed architecture docs. The Marketing System's planner run proved that if the docs are implementation-ready, going straight to Claude Code saves hours.
